  If the exact NS31A signal does not exist in Ibex, find the semantically
  closest equivalent. If no single signal maps directly, combine signals
  that together capture the same security guarantee.

================================================================================
TRANSLATION RULES — MANDATORY — NEVER VIOLATE ANY RULE
================================================================================
SIGNAL RULES:
  R1. Use ONLY signal names from AVAILABLE SIGNALS. Zero exceptions.
  R2. Signal widths must match — check the width column before using a signal.
  R3. Never concatenate or slice signals unless required to match a known Ibex
      bit range.
  R4. For enum-typed signals: use symbolic constant names from PACKAGE TYPE
      DEFINITIONS above. Never hardcode raw bit values (e.g., 2'b11).
  R5. For struct-typed signals: use dot notation with exact field names from
      PACKAGE TYPE DEFINITIONS above. Never invent or guess field names.

TRANSLATION APPROACH RULES:
  R6. NEVER skip an assertion group. Every group must produce at least one
      SVA property in the bind file.
  R7. If the NS31A signal name does not exist in AVAILABLE SIGNALS:
        - Find the semantically closest signal in AVAILABLE SIGNALS
        - Translate the security intent using that signal
        - Document the mapping in the JSON log
  R8. If an assertion checks a property that Ibex expresses differently
      (e.g., NS31A has an explicit status flag that Ibex derives implicitly):
        - Use the Ibex signals that together encode the same condition
        - Write the assertion in terms of those Ibex signals
  R9. If a NS31A property references architectural state that Ibex handles
      internally (not observable as a port or internal signal):
        - Express the OBSERVABLE CONSEQUENCE of that property in Ibex
        - Example: "CSR write blocked when locked" → check that the relevant
          output signal is unchanged when the lock condition is active

SVA SYNTAX RULES:
  R10. Every property block: @(posedge {{CLOCK}}) disable iff (!{{RESET}})
  R11. Temporal delay: use ##N (N >= 1). Never use #delay.
  R12. Previous cycle value: use $past(signal) or $past(signal, N).
  R13. Implication: use |-> (overlapping) or |=> (non-overlapping) correctly.
       |->  : consequent starts at same clock cycle as antecedent end
       |=>  : consequent starts one cycle after antecedent end
  R14. Every property must have both a named property block AND a named assert:
         property PROP_NAME; @(posedge {{CLOCK}}) disable iff (!{{RESET}})
           <antecedent> |-> <consequent>;
         endproperty
         assert property (PROP_NAME);
  R15. Property names: {{MODULE_SHORT}}_SEC_<N> where N is a sequential integer.
  R16. Antecedent must be reachable in normal (non-Trojan) operation.
       If the antecedent can never fire on clean RTL, rewrite it.
  R17. No $display, $error, or simulation-only constructs inside properties.

BIND FILE PORT RULES:
  R18. ALL ports in the assertion module must be declared as INPUT.
       This includes signals that are outputs of the DUT.
       The assertion module only observes signals — it never drives them.
       Driving a DUT output from the assertion module causes a driver conflict.
  R19. Clock and reset must be in the port list as input.
  R20. Bind statement goes OUTSIDE and AFTER the assertion module:
         bind {{MODULE_NAME}} {{MODULE_NAME}}_assertions u_{{MODULE_SHORT}}_assert (.*);
  R21. The (.*) wildcard connects all ports by name — valid only if port names
       in the assertion module exactly match DUT port names.
